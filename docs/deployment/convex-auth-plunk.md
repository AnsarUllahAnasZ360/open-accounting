# Convex Auth and Plunk Setup

Convex Auth is configured with a custom email provider named `plunk`.

## Required Convex Deployment Environment Variables

- `SITE_URL`: the public app URL used in magic links, for example `https://accounting.zikrainfotech.com`
- `JWT_PRIVATE_KEY`: generated by the Convex Auth setup command
- `JWKS`: generated by the Convex Auth setup command
- `PLUNK_API_BASE_URL`: `https://api.plunk.zikrainfotech.com`
- `PLUNK_SECRET_KEY`: server-only Plunk secret key
- `PLUNK_FROM_EMAIL`: verified sender email address
- `PLUNK_FROM_NAME`: optional sender display name

## Required Vercel Environment Variables

- `NEXT_PUBLIC_CONVEX_URL`
- `CONVEX_DEPLOYMENT`
- `NEXT_PUBLIC_APP_URL`
- `NEXT_PUBLIC_APP_NAME`
- `NEXT_PUBLIC_PLUNK_PUBLIC_KEY`

## Notes

Classic Convex Auth is still beta and its Next.js server-side helpers are still a caveat. The first app version keeps the app shell on Convex's plain reactive provider while the backend auth routes, keys, and Plunk sender are already configured. The next auth slice should add the sign-in screen and client-side auth provider deliberately, then gate Convex queries in client components. Server-side auth and middleware can be added later after the dashboard workflow is stable.

Magic-link emails must not include financial or accounting data. The email should only identify the product and carry the sign-in URL.
