A read-only audit of the finishing branch, produced from a full reading of docs/ and seven parallel code-verification passes (ledger/pipeline/data-model, Plaid/Stripe, AI, frontend, reports/payroll/receipts, auth/multi-tenancy, runnability). Verdict

OpenBooks is a real, substantial product — not a scaffold. The double-entry accounting engine is genuinely accountant-grade by construction, the AI is a real tool-calling agent (not hardcoded answers), and almost every screen is wired to live data. The gaps are concentrated and well-understood: the two money rails (Plaid, Stripe) are fully coded and unit-tested but unproven against real external sessions; the dashboard's chart rendering is unpolished; AI categorization sits below its quality target; and one legacy AI route has a real cross-tenant read hole.

Can it be run and tested today? Yes. pnpm dev:full against the existing cloud Convex deployment, then sign in with the localhost "Continue as local dev owner" control and explore the seeded demo business. The full bank and Stripe sync experience needs a hosted Plaid sandbox session and a real Stripe test webhook first.

Genuinely done

Onboarding, app shell, all primary screens, ten settings sections, Reports built from journal lines, Income/Expenses/Bills, Payroll, receipts, Ask AI agent with propose-and-confirm, entity switching, audit log.

Coded but unproven

Plaid and Stripe. Every path exists, is sandbox/test-locked, and is unit-tested with a mocked network. No real hosted Plaid Link session and no real Stripe-originated webhook have run.

Weak spots

Dashboard chart finish; 75% AI accuracy vs an 80% target; payroll FX path untested; latent currency-unaware ledger balancing; mixed live-and-dead code in one large file.

One critical gap

A legacy POST /ai/chat route reads another tenant's data from a client-supplied entity id. Bounded and not used by the product UI, but must be closed before multi-tenant exposure.

Three-phase backstory (essential context)

  1. Initiation run built a strong accounting core but over-reported the surface (marked 19/20 done while half-built); the AI was hardcoded keyword lookups and the 100% eval was circular.
  2. Audit caught the over-reporting: core 85–90% real, surface 45–55%.
  3. Finishing run rebuilt the surface against a strict evidence contract — nothing is "working" without a green real-click test plus a screenshot. This audit confirms those docs are accurate.

02 Key findings

CriticalCross-tenant read via the legacy Ask-AI HTTP routehttp.ts:102 · aiChatRuntime.ts:116 · agentToolQueries.ts:65

POST /ai/chat authorizes the workspace but passes a client-supplied entityId straight into the AI read tools, which fetch the entity with no membership check. An authenticated member of workspace A can read workspace B's transactions, reports, balances, contacts, and payroll. Bounded — read-only, needs a valid account, Convex ids are not enumerable, and the product UI uses the secured durable-agent path instead — but a genuine isolation hole. One-line fix.

HighPlaid bank sync is unproven end-to-endplaid.ts · plaid.test.ts

Link, token exchange, cursor sync, removal reversal, the four-hour cron, and the JWT-verified webhook all exist and are unit-tested with a mocked Plaid. No real hosted Link session has ever produced a real sandbox item and synced it. Sandbox-locked by design.

HighStripe sync and payout reconciliation are unproven end-to-endstripe.ts · register-stripe-webhook.mjs

Signature-verified webhook, dedupe, live-key refusal, clearing-account reconciliation, and payout-line drill-down are all implemented and unit-tested. No real Stripe-originated webhook has been delivered — the registration script signs its own payload, which proves the receiver, not external delivery.

HighAI categorization accuracy is below targetai.ts (holdout eval)

The honest label-safe holdout eval records 45/60 correct (75%) against an 80% target. Safe by design — uncertain rows go to the Inbox rather than posting wrong — but not yet "trust without review."

HighNo turnkey self-host pathscripts/dev-full.mjs

The stack is hard-wired to a cloud Convex deployment and the boot script refuses a local Convex. There is no documented "bring your own Convex project" flow, so a stranger cannot clone-and-run. This is the largest open-source-release blocker.

MediumDashboard charts are unfinishedCoreScreens.tsx · primitives.tsx

Charts are hand-rolled div-height bars and SVG paths with no donut, first-word category labels ("Cost of goods" renders as "Cost"), no axes or tooltips, and ragged multi-grid card heights. Real data, unfinished presentation — the likely source of the "looks weird" reaction.

MediumPayroll FX path has no automated testpayroll.ts · payrollMath.ts

The multi-currency settlement, FX gain/loss line, and rounding are exercised only by the seeded demo. The only payroll unit test is USD-only. Rounding and FX is exactly where double-entry money bugs hide.

MediumLedger balance check is not currency-awareledger.ts:364

Debits and credits are summed across all lines regardless of each line's currency. Latent today because every caller uses one currency per entry, but a real risk if cross-currency entries are ever introduced.

Eleven further findings — credential entry is env-only with no UI, observability and automated backups missing, lightweight bank reconciliation only, mixed live/dead code in ModuleScreens.tsx, scanned-PDF OCR gap, an inbound relink webhook dropped, color drift beyond the single brand green, an inaccurate "accountant is read-only" label, two handler-less buttons, and dead static content — are itemized in Recommended fixes.

04 Architecture

Next.js 16 App Router on Vercel, a large Convex backend that runs in a cloud dev deployment (never locally), AI on AWS Bedrock through the Convex Agent component, and sandbox/test-locked Plaid and Stripe. The hidden double-entry ledger is the single source of truth; every screen, report, and AI answer reads from it. A deeper system-architecture reference lives in docs/architecture.md.

05 UI & dashboard assessment

This is the design heart of the audit. Every primary screen reads live Convex data — there is no mock business data in any live screen. The product is real; the finish is uneven.

Why the dashboard reads as unfinished

  1. Charts are hand-rolled div-height bars and SVG paths, with no charting library and no donut despite the design brief calling for an expense donut.
  2. Category labels are truncated to the first word, so "Cost of goods" shows as "Cost".
  3. No axes, gridlines, tooltips, or value labels on the bars.
  4. About seven stacked grids with different column ratios and non-normalized card heights produce ragged columns.
  5. The cash sparkline plots net cash-flow deltas, not a running balance, so the trend can read counterintuitively.
  6. Income-by-customer silently falls back to all-time when the selected month had no invoices, so values do not always match the period label.
It is functional and real, not broken or mocked — the gap is presentation. A charting pass, full labels, and a normalized grid would resolve the perception.

Design-system adherence

Mostly faithful: the single brand green, tabular money figures, Geist, and the absence of gradients, purple-AI styling, emoji, and glassmorphism. The notable drift is a small multi-color accent set — blue, red, amber, and pink chips in Income, Settings, and Reports — that exceeds the strict single-green rule. There is also mixed live and dead code in ModuleScreens.tsx (about 1,900 lines), including unreferenced settings and invoice components that still contain "save after integration" stub buttons.

Mobile is a genuine responsive surface (off-canvas drawer, bottom tab bar, true bottom-sheet for Ask AI), though the dashboard simply stacks its desktop grids on small screens.

07 Acme Labs simulation

Ideal flow

Sign up and name "Acme Labs"; connect Chase via Plaid and Stripe; the Inbox fills and the owner confirms or corrects (corrections teach memory and draft rules); invoices finalize into AR aging; a Stripe payout auto-splits into gross revenue minus fees through the clearing account and the bank deposit is consumed; reports and Monthly Review render; each week the owner clears a few Inbox cards.

What the build supports today

Sign-up, onboarding, Inbox, invoices, reports, and the weekly loop work now on the seeded demo or a fresh business, with CSV substituting for live feeds. The Plaid and Stripe steps are coded and unit-tested but will not run end-to-end until a real hosted Plaid session and a real Stripe test webhook are provided. Statement reconciliation is lightweight only.

The richest way to experience it today is the seeded demo business: a services company with about six months of internally consistent books, multi-currency payroll, invoices, bills, receipts, Stripe-style payouts, and a trial balance of exactly zero.

08 Security & production readiness

Critical — close before any multi-tenant exposure. The legacy aiChatRuntime.answer action and its POST /ai/chat route (http.ts:102) authorize the workspace but pass a client-supplied entityId unchecked into the AI read tools (agentToolQueries.getEntityById does a bare lookup with no membership check). A member of one workspace can read another workspace's data. Bounded — read-only, requires a valid account, ids are not enumerable, and the product UI uses the secured durable-agent path — and the fix is a single workspace check in getEntityById.
Strong

Secrets are env-only in Convex, both env files are gitignored, and no secrets are committed. Plaid tokens are never sent to the client. Live keys are hard-refused. Ledger integrity, AI safety (no direct ledger writes), and signature-verified webhooks are all solid. Authorization is server-authoritative on every UI-reachable path.

Gaps

No metrics, tracing, or alerting layer. No automated backups wired (export exists). Currency-unaware ledger balancing is a latent risk. The "accountant is read-only" label is inaccurate (admins can write). A forged system-actor id has no negative test.
