## Authorization coverage matrix (E14-T5)

**Rule audited.** Every exported Convex query / mutation / action must re-check
workspace/entity authorization on the server (AGENTS.md), so a caller can never
read or write another LLC's books. This is the core defense against cross-tenant
data leakage before public launch.

**How this matrix is produced.** `scripts/authz-coverage.mjs` enumerates every
exported function across `convex/*.ts` and classifies each as:

- **authorized** — the function body (or a same-file helper it calls, resolved
  one level deep) calls a server-side authorization guard from `convex/authz.ts`
  (e.g. `requireWorkspaceRole`, `requireWorkspacePermission`, `requireUserId`,
  `requireAnyWorkspaceRole`, `getEntityForRead`/`getEntityForWrite`,
  `authorizeThreadAccess`, or a per-module guard that wraps them such as
  `requireTransactionForAdmin`, `authorizeCategorizationRead`,
  `authorizeEntityForConnections`); OR it calls a reviewed cross-file guard
  helper (`resolveActiveEntity` / `resolveDefaultEntity` / `runWorkspaceReset`,
  each of which itself calls a guard); OR it is an action (or same-file helper)
  that delegates to a guarded query/mutation via `ctx.runQuery` /
  `ctx.runMutation` / `ctx.runAction`.
- **intentionally-public** — anonymous-by-design public surface, explicitly
  allow-listed with a rationale.
- **FINDING** — a NEW exported function that is neither guarded nor
  allow-listed. The script **exits non-zero** on any FINDING, so CI fails the
  moment a new exported function ships without a guard and without being
  triaged. There are currently **zero** findings.

```
authz-coverage: 363 exported functions — 357 authorized, 6 intentionally-public, 0 FINDING
```

Run locally:

```bash
node scripts/authz-coverage.mjs            # summary + non-zero exit on any FINDING
node scripts/authz-coverage.mjs --markdown # regenerate the table below
```

### Triage of functions without a direct guard call

The script flags functions whose body has no direct guard. Each was manually
triaged; all resolved to **authorized** or **intentionally-public** (no real
findings):

- **Genuinely public (anonymous by design):** `requestAccess.submit` (landing
  "request access" form — write-only, email-validated) and `team.lookupInvite`
  (an invitee with no session looks up a pending invite by a 32+ char
  unguessable token; the token is the capability and short/unknown tokens return
  `invalid`).
- **Ask-AI agent read tools** (`agentToolQueries.*ForEntity`,
  `reportViews.reportPackForEntity`, `aiThreads.threadContext`): `internalQuery`
  functions reached only from the streaming action **after**
  `authorizeThreadAccess` resolves the thread's owning entity. The
  thread-ownership row is the authorization boundary (documented inline in
  `reportViews.ts`); these are never client-callable.
- **Connections / Plaid / Stripe internals** (`connections.*`, `plaid.*` sync
  internals, `stripeWebhook.recordEvent`): invoked only by guarded connection
  mutations, the sync cron, or the **signature-verified** Stripe/Plaid webhook
  handlers — never directly by a client.
- **Demo seeding + real-test reset job rows** (`seedDemo.*`, `realTestReset`
  job-row writers): `internal*` writers driven by the guarded seed action or the
  `workspace.reset`-gated rebuild action.
- **Categorization eval/batch internals** (`ai.holdoutTransactionResult`,
  `ai.recordHoldoutCategorizationEvalRun`,
  `bedrockCategorizer.categorizePendingTransactions`,
  `semanticMemory.proposeCategorizationMemory`): the public categorize action
  delegates through `runCategorizationBatch` to
  `internal.ai.categorizationBatchCandidates`, which guards via
  `authorizeCategorizationRead`; the rest are `internal*` and reached only from a
  workspace-guarded action.
- **Owner bootstrap** (`authAdmin.ensureOwnerWorkspace`): `internalMutation` run
  from the auth callback to materialize the configured owner's workspace.

### New function classes for this sprint

Two classes the ticket calls out explicitly — both have now landed and are
classified in the matrix:

- **Unified `credentials` read/write/resolver (E3).** The unified `credentials`
  table is live. The client-facing writers/readers are workspace-guarded and
  appear as `authorized`: `credentials.saveCredential`,
  `credentials.deleteCredential`, `credentials.credentialStatus` (workspace-scoped
  for AI/Plunk, per-entity for Stripe). The resolver internalQuery
  `credentials.getActiveCredential` is reached ONLY via
  `ctx.runQuery(internal.credentials.*)` from the guarded AI/Plaid/Stripe/Plunk
  resolvers + the signature-verified Stripe webhook, and is allow-listed
  `authorized` (not client-callable). A NEW exported credential function would
  flag as a FINDING until it carries a guard or is triaged.
- **Shared no-login demo read path (E11).** `demo.demoView`, `demo.demoContext`,
  and `demo.demoDashboard` are **intentionally-public but read-only**: the demo
  workspace is resolved entirely server-side via the registry
  (`demoKind === 'public'`), never from a client-supplied id, and can only ever
  return the single public demo workspace. There is no demo mutation; demo writes
  are blocked by `assertNotDemoWrite` on every workspace-scoped mutation. These
  three reads are allow-listed `intentionally-public`.

### One real finding fixed this pass (E14-T5)

The coverage re-run after the codebase grew (E2/E4/E9/E11/E13) surfaced one
genuine gap: `aiSdkRuntime.testProviderConnection` (and its `ai.testProviderConnection`
wrapper) accepted a client-supplied `workspaceId` and resolved + exercised THAT
workspace's saved AI credential **without re-checking the caller's membership** —
a signed-in user could probe another workspace's provider readiness. Fixed by
re-checking workspace membership server-side at the start of the action via
`ctx.runQuery(internal.aiThreads.assertWorkspaceMember, { workspaceId })` before
any credential read or provider call. (The shared `resolveActiveAiModel` resolver
stays intentionally unguarded so the internal categorize/CFO/chat runtimes can use
it without a user identity; the membership guard lives on every client-facing
entrypoint instead.) `convex/authzCoverage.test.ts` covers the anonymous-rejection
path; the cross-workspace path is now membership-gated.

### Automated rejection tests

`convex/authzCoverage.test.ts` proves a representative function per category — a
view query, a ledger mutation, an action, and a connections/settings mutation —
(a) rejects an anonymous caller with `OpenBooks requires sign-in`, and (b)
rejects a member of workspace A targeting an entity in workspace B
(cross-tenant). Green under `pnpm test:unit`.

## Secret handling & encryption-at-rest (E14-T6)

**Unified `credentials` store.** All four BYO secret kinds — `ai`, `plaid`,
`stripe`, `plunk` — converge on one `credentials` table (`convex/credentials.ts`,
`convex/schema.ts`). The old per-field `aiCredentials` shape is collapsed into
this single encrypted-blob model:

- **Write (`credentials.saveCredential`, mutation).** Re-checks workspace authz
  (`requireWorkspaceRole(workspaceId, "admin")`) and `requireSecretVault()`
  (the encryption key must be present) *before* touching any secret. The whole
  payload is JSON-serialized and encrypted with `encryptSecret` into a single
  `encryptedPayload` column. A plaintext secret is **never** written: if
  encryption returns null the mutation throws rather than fall back to plaintext.
  Scope rules are enforced server-side — AI/Plunk are workspace-scoped; Stripe
  requires an `entityId` that belongs to the workspace (per-business).
- **At rest.** `encryptedPayload` is AES-GCM ciphertext keyed by a 32-byte key
  **derived via HKDF-SHA-256** from `OPENBOOKS_SECRET_ENCRYPTION_KEY`
  (`convex/secretBox.ts`), with a per-ciphertext random 16-byte salt and 12-byte
  IV (v2 format). HKDF (RFC 5869) is the correct KDF: it extracts uniform key
  material and expands to exactly 32 raw bytes, unlike a bare SHA-256 digest
  (the prior v1, kept decrypt-only for upgrade). Domain-separation `info`
  (`openbooks/secretBox/v2`) prevents cross-purpose key reuse.
- **Read surface (`credentials.credentialStatus`, query).** Workspace-guarded
  (`requireWorkspaceRole("member")`) and returns **only** safe metadata:
  `keyPreview` (masked), `status`, `lastValidatedAt`, presence booleans
  (`hasApiKey`/`hasAwsKeys` derived from presence, never the value), plus
  non-secret config (`baseUrl`/`region`/`fromEmail`/`fromName`/`model`). The raw
  `encryptedPayload` is never selected into the return shape.
- **Resolver (`credentials.getActiveCredential`, internalQuery).** The only path
  that decrypts is `getActiveCredential` + the per-provider resolvers, which are
  `internal*` — reached only via `ctx.runQuery(internal.credentials.*)` from the
  guarded runtimes and the signature-verified webhook. The decrypted secret never
  crosses the client boundary.
- **Never logged.** The save audit row (`auditEvents`) records only
  `credential.<kind>.saved` + a non-secret summary. Thrown provider errors are
  passed through `secretRedaction.safeErrorMessage`, which redacts every env
  secret and the resolved credential's key material before the message can reach
  a client. `convex/secretSafety.test.ts` asserts no integration query returns a
  ciphertext/raw-secret field and that the redaction helper strips planted keys.

**Live connectors local (decided).** There is **no** "sandbox/test keys only"
guard. Live Plaid (development/production) and live Stripe keys are permitted for
local + self-host use; the posture is *secrets encrypted at rest* + *live-key
shape validation*, not *live keys banned*:

- Encryption-at-rest is enforced on the live Plaid path (`convex/plaid.ts`,
  access tokens encrypted via `secretBox` before persistence).
- Live Stripe Connect requires a real live key shape: `connections.ts` rejects a
  `live` mode credential unless it starts with `sk_live_` (and `test` unless
  `sk_test_`), so a mislabeled key can't be stored against the wrong mode.

## Webhook signature verification (E14-T6)

Both inbound webhooks verify signatures before any side effect; an invalid or
absent signature returns **400 with no DB write**.

- **Stripe** (`convex/http.ts` `/stripe/webhook` → `verifyStripeWebhookSignature`
  in `convex/stripeWebhook.ts`): parses the `Stripe-Signature` header, enforces a
  replay-tolerance window (`Math.abs(now - timestamp) > tolerance` → reject,
  default 300s), recomputes the HMAC over the raw body, and compares with a
  **timing-safe** equality (`timingSafeEqual`). Event-id dedupe lives in
  `stripeWebhook.recordEvent` (written only after a verified delivery). A
  signed-but-unverifiable delivery flags the matching connected account's webhook
  as `failing` (best-effort) and returns 400. Because a live Stripe connection is
  only marked `listening` by a **real verified delivery** (`markStripeWebhookDelivery`
  on outcome `verified`), the verified-webhook precondition is enforced.
- **Plaid** (`convex/http.ts` `/plaid/webhook` → `verifyPlaidWebhookSignature` in
  `convex/plaidWebhook.ts`): validates the JWT verification header (ES256 only),
  fetches the key by `kid`, verifies the JWT, checks the body SHA-256 matches
  `request_body_sha256`, and rejects a **stale** signature (`iat` older than the
  5-minute window). Failure returns 400.

**Tamper-rejection tests (present, green).** `convex/stripeWebhook.test.ts`
asserts a changed payload AND a stale timestamp are both rejected; the valid path
verifies. `convex/plaidWebhook.test.ts` exercises the signed-hash verification
path. These satisfy the DoD's "tampered/absent signature rejected" requirement.
